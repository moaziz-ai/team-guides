Your OpenClaw system

The complete operating manual for your AI team, the infrastructure that runs it, and how to interact with all of it day to day.

Version 1.0 · Last updated 19 Apr 2026 · Based on Karpathy's LLM Wiki pattern

The big picture

14 agents split across three pods (Sooner, Axeed, Personal). A handoff layer connecting them to Claude Code for heavy work. Memory that compiles and validates itself. Daily GitHub backups. All running on your Mac.

Core insight: OpenClaw handles conversation and orchestration. Claude Code handles the heavy thinking. Memory lives in markdown files that compile and propagate rules automatically. Nothing is a chatbot - everything is a durable system.
Technical foundation
Architecture is an implementation of Andrej Karpathy's "LLM Wiki" pattern: raw notes → compiled summaries → validated → promoted → re-loaded. Memory lives in markdown on disk. Agents read the compiled layer first, raw layer only on demand. Periodic "dream cycles" detect patterns and promote stable knowledge to the canonical wiki.

Stand-ups are operational only. They summarize yesterday, flag today's priorities, and surface any decisions needed from Mo. They do not modify the wiki or protocol files.

Reading discipline: each stand-up is designed to read in under 60 seconds. If they grow longer, it's a signal something is wrong in the synthesis.
Technical detail
Triggered by launchd jobs (ai.openclaw.standup.{lisa,elon,iris}). Each trigger runs openclaw-delegate to synthesize via Claude Code (cheap), then makes a small OpenRouter call via the agent to post the result to its configured surface.

Weekly dream cycles

Sunday 7:00 PM Dubai. Two triggers fire:

AgentSurfaceScope
LisaSlack #sooner-ai-leadership (team visible)Sooner weekly dream
ElonTelegram DM to MoAxeed weekly dream

Dream cycles synthesize the week and produce:

  • Patterns observed - rules Mo repeated, agent convergence
  • Auto-promotions - high-confidence rules written directly to the wiki
  • Proposals awaiting approval - medium-confidence items numbered for Mo to approve or reject
  • Stale or contradictory items - things in memory that look wrong
  • Agent health - who was active, who was idle
Auto-promotion policy: High-confidence rules (said 3+ times across different contexts) get written to the wiki automatically. Medium-confidence items need your explicit approval. DECISIONS.md and protocol files never auto-promote.

Monthly cross-pod review

1st of every month, 9:00 AM Dubai. Iris runs a cross-pod dream that sits on top of Lisa's and Elon's weekly ones.

She covers:

  • Patterns appearing in both pods that should be cross-pod standards
  • Agents in different pods stepping on each other's scope
  • System-level patterns and health
  • Role boundary issues

Cross-pod rules never auto-promote. Always proposed, never written without your approval.

Memory architecture

Five layers, ordered from raw to canonical:

1. Agent daily notes Per-agent · Raw

Raw daily logs written by each agent during work. Personal, ephemeral, feeds into higher layers.

~/.openclaw/agents/*/memory/YYYY-MM-DD.md

2. Agent MEMORY.md Per-agent · Curated

Durable memory per agent. How that agent operates. Updated when something stable emerges.

~/.openclaw/agents/*/MEMORY.md

3. Shared pod current truth Pod-wide · Live

What is happening right now at the pod level. Refreshed as state changes.

~/.openclaw/shared-memory/{sooner,axeed,personal}/STATUS.md
~/.openclaw/shared-memory/{sooner,axeed,personal}/DECISIONS.md
~/.openclaw/shared-memory/{sooner,axeed,personal}/PROJECTS.md

4. Wiki Pod-wide · Canonical

Canonical truth. Brand, product, architecture, fundraising. Changes only via dream cycles with your approval.

~/.openclaw/wiki/{sooner,axeed}/

5. Protocol files Functional playbooks

Long-form operating playbooks:

  • Draper's BRAND_PROTOCOL.md
  • Reid's FUNDRAISING_PROTOCOL.md
  • Nora's SUPPORT_PROTOCOL.md
The promotion flow (Karpathy compound loop)
Agent activity → daily notes (raw) → Sunday dream review → classification (auto/propose/info) → high-confidence writes directly to wiki → medium-confidence proposals wait for approval → approved writes happen → next session inherits updated wiki via CLAUDE.md pointers.

The Claude Code handoff system

Auto-compile
After every substantive Claude Code session (default: 30+ new log lines), a Python double-fork daemon spawns a detached claude process that reads the raw live-status log plus the project files and writes a compiled PROJECT_SUMMARY.md. Next session opens with the compiled state loaded first, raw log only when needed. Scales to hundreds of sessions without the log becoming unreadable.

The delegate pattern (cost savings)

OpenClaw agent turns use OpenRouter credits (real per-token cost). Claude Code uses your Claude Max subscription (fixed monthly, effectively free within cap). The delegate pattern offloads heavy work from OpenRouter to Claude Code.

When to delegate

Delegate for:

  • Drafting long emails (investor updates, contract reviews)
  • Memory synthesis (dreams, monthly reviews)
  • Mass operations (triage 50 emails, review 20 PRs)
  • Research passes (reading many docs, producing reports)
  • Anything requiring more than ~1000 tokens of context

Don't delegate for:

  • Conversational replies (overhead ruins the live feel)
  • Short tasks under 200 words
  • Time-sensitive acknowledgments
Estimated savings: $100 to $250 per month by offloading all synthesis tasks. Dream cycles alone save $50 to $100 per month at typical usage. 
openclaw-delegate \
  --agent nora \
  --task "Triage these 30 emails. Classify by urgency, propose reply drafts..." \
  --context ~/nora-inbox-dump.md \
  --output ~/nora-triage-result.md

Backup and recovery

What is backed up

Everything in ~/.openclaw/ except the excluded items below. 674 files on initial push, auto-synced daily.

What is NOT backed up (intentionally)

  • .env (Slack/Telegram/OpenRouter tokens) - store in 1Password
  • browser/ (Chrome profiles, cookies)
  • credentials/, identity/
  • */sessions/*.jsonl (conversational transcripts)
  • Delegation temp folders
  • Caches, logs, binaries over 100MB

Recovery path if your Mac dies

  1. Install OpenClaw, Claude Code, Node, gh CLI on the new machine
  2. Clone github.com/moaziz-ai/openclaw-config to ~/.openclaw/
  3. Restore .env from 1Password
  4. Re-authenticate integrations: gog auth, op signin, gh auth login
  5. Run openclaw gateway restart
  6. Your agents come back with full memory intact

  • Heavy synthesis requests auto-route to Claude Code via delegate

  • Read two weekly dream summaries
  • Approve/reject medium-confidence proposals
  • High-confidence auto-promotions already done

Monthly

  • Iris's cross-pod dream arrives on the 1st
  • Approve any cross-pod rule changes

Operating principles

1. Don't micromanage. If you repeat a rule 3+ times, the dream cycle auto-promotes it. You don't need to manually add it to the wiki.
2. Use delegate for heavy work. Agents will offload long synthesis to Claude Code. This saves real money.
3. Trust the hooks. Open Claude Code in a project folder, CLAUDE.md auto-loads. Don't re-brief manually.
4. Approvals are async. Agents don't chase. Reply on your own timeline.
5. When in doubt, ask Iris. She owns the infrastructure. If something feels off, tell her.
